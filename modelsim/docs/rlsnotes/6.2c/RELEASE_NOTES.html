    • Key Information

    • The automatic invocation of the design-wide performance optimization tool "vopt" has been enabled. Upon invocation, the simulator determines if vopt has been run on the top-level module(s) you supplied; if not, the simulator invokes vopt and outputs a message similar to:
      # ** Note: (vsim-3812) Design is being optimized...
      The optimization step will improve run time performance and affect the visibility of objects in the design. To enable debug visibility it is recommended you add additional options to vopt on the vsim invocation via the -voptargs argument. For example: vsim -voptargs="+acc" which enables full debug access to objects in the optimized design. To understand the full usage of +acc and -voptargs, please refer to the chapter "Optimizing Designs with vopt" in the User's Manual for more information.
      If required you may obtain the previous behavior of vsim, where it does not perform performance optimizations, by setting the modelsim.ini variable "VoptFlow" to 0 (zero). An alternative to setting VoptFlow to zero, where interactive debug compile throughput is required you can use the -novopt option on the vsim and compilation command line (vcom/vlog).

        Solaris 10 has the following limitations:
      • In the OS, vt alarm is producing irregular and random beats between 20ms and 100ms. As a result, the profiler produces the error message, "Too few samples." To workaround this problem, add the following line to the file /etc/system and reboot the system:
        set hires_tick=1
        This will produce consistent sample times. The minimum sample time will be 20ms. This problem has been filed as Sun CR 6290459 and will be fixed in a coming kernel patch. With the patch, the minimum sample time will be approximately 10ms, as in previous Solaris releases.

    User Interface Defects Repaired in 6.2c

    • Accessing dynamic local variables from the Objects window caused a crash in certain cases. Dynamic locals will no longer appear in this window, now only static locals are displayed.
    • Selection of a non-active process in the Structure or Dataflow window in some cases required excessive time when the Active Process window was populated. An example of this scenario is immediately after elaboration but before the first run command and when many Verilog "initial" processes were scheduled. This operation now runs significantly faster.
    • If the -pslfile switch was used before the -cover switch on the vlog command line, there was no code coverage in vsim in vopt mode.
    • vsim silently ignored invalid switch options that began with -a, -k, or -0. If a valid switch starting with one of these letters or numbers was misspelled, the simulator did not report an error, nor was the switch recognized.
    • The option of specifying a dataset name for opening vsim in coverage view mode did not work correctly in 6.2b. Here is an example of the invoking vsim in coverage view mode:
      vsim -viewcov dsname=file.ucdb
    • The Library view did not update correctly when a change directory was performed, or when a project was closed or a new one opened. Some of the logical library mappings were incorrectly retained.
    • The add wave -divider command in some cases caused internal memory to become corrupt and subsequent commands behaved incorrectly or stopped working.
    • The coverage report did not show exclusions which were specified using line ranges.
    • Command completion for the coverage command was incorrect in vsim and viewcov modes.
    • Coverage exclude command doesn't update GUIs in viewcov mode.

    Verilog Defects Repaired in 6.2c

    • Design optimization failed in some cases when inlining modules with SystemVerilog var port connections.
    • Incorrect compilation errors resulted in certain cases for an assignment from a class object to a base type variable when the object type was derived from a base type through multiple parameterized class types.
    • Incorrect errors regarding the context of genvar use could be reported when optimizing a design containing generate for loops.
    • System functions $sscanf and $fscanf continued scan of their inputs even if no acceptable characters were found for a conversion specification. Scan is now terminated in this case.
    • The conversion specification "%f" in system functions $sscanf() and $fscanf() sometimes allowed conversion of an illegal sequence of characters.
    • Compiling two source files that declare the same module name in some cases resulted in the first file name being reported as the source of an error or warning detected while compiling the second file.
    • Under certain conditions, the gate-level optimized cell evaluator ignored notifier register toggles fanning into always blocks.
    • When the vsim command line options +multisource_int_delays and/or +transport_int_delays were specified with the vsim command line option -v2k_int_delays or compiled SDF files, in simulating a design that had nets with large fanin and fanout, the SDF load times were very long.
    • When +grp_ntc_outputs was specified for vlog and +multisource_int_delays or +transport_int_delays was specified for vsim, some simulations looped indefinitely.
    • Design optimization failed in certain cases of self-recursive module instantiations.
    • An error regarding "too many indices" resulted when indexing into a class property of a parameterized class where the type of the property was a type parameter which resolved to an array type.
    • When specializing a parameterized class, errors were not reported when specifying too many parameters, when associating to an incorrect name, or when providing multiple definitions of a parameter.
    • Calling a task (residing in an interface) through a virtual interface resulted in an erroneous error.
    • Elaboration warnings resulted when referencing a static property in an imported class where the class name and package name were the same.
    • Errors resulted when parameterized class specializations were differentiated in source only by an enum type versus an integral type. Such specialized types were incorrectly treated as the same type.
    • Null values, "()", in IOPATH and SETUPHOLD statements, were not correctly annotated to optimized cells.
    • SystemVerilog packages did not correctly scale time values according to the current `timescale directive or timeunit/timeprecision declarations.
    • An internal error in vlog occurred if a randsequence statement was specified within the body of an out-of-block class method.
    • Misleading 'Missing connection for port ...' warnings could have occurred when a 'Too few port connections' situation existed and the code contained named port association. The number of 'Missing connection for port ...' warnings will now match the number of missing ports in the 'Too few port connections' warning message.
    • When a SystemVerilog module containing a package import was optimized by vopt it would sometimes generate error messages like: ** INTERNAL ERROR: export_lookup() failed.
    • When compiling SystemVerilog code with -mfcu the command line switches following -mfcu were being ignored by vopt.

    PLI Defects Repaired in 6.2c

    • PLI calls to acc_next_port failed in certain cases when accessing memory beyond the size of an internal data structure when accessing a bit net.

    VHDL Defects Repaired in 6.2c

    • A wait statement with a function call in the condition clause (i.e., until condition) may not have executed correctly.
    • If a library, entity, and architecture had the same simple name and you tried to attribute the architecture, an error occurred. In this case the following error message was generated:
      ** Error: src/src_file.vhd(21): 'library_entity_architecture_name' is not in the specified attribute entity class. Possible ambiguous name.
    • In some cases an event on a slice of a port failed to trigger a process that was sensitive to that slice. As a result, incorrect simulation results occurred.
    • In some cases involving disconnection of a port, incorrect error messages were produced when the formal port was guarded and the actual port was not guarded.
    • When running vopt or vsim, if -L directives are present when optimzing the design, errors occurred if the library names conflicted with other VHDL identifiers. When this occurred error messages about the name being of the incorrect class were generated.
    • Using an alias to a subprogram declaration in a package header and then attempting to define the subprogram body corresponding to that alias is detected as compilation error.

    Mixed Language Defects Repaired in 6.2c

    • The error message:
      ** Error: (vsim-3171) Could not find machine code ...
      was issued by vsim in situations where a module instantiates both Verilog modules containing instantiations of VHDL architectures, and direct instantiations of VHDL architectures. The VHDL architectures had to have been specialized, in that they contained generics inferred to be constants by vopt.

    General Defects Repaired in 6.2c

    • The vopt -o argument allowed characters in the name that caused vsim to be unable to simulate the result. Now only alphanumeric and underscore characters are allowed in the name.
    • vcd dummports failed to capture net values from the force command, force -freeze command, and the Verilog force statement.
    • vopt failed to find a library in some cases in which,
      • the physical library name contained upper case letters,
      • no library mapping was given,
      • the library name was specified using the -L or -Lf command line options.
    • The tool vcd2wlf rarely created incorrect results in the case where a scalar is an alias (same VCD id) with a member of a bus.
    • An erroneous out-of-bounds error could be reported during simulation on linux, linux_x86_64, or win32 platforms for operations (e.g. assignment) on character or byte-sized data.
    • Condition and expression coverage tables were not always built correctly. If an expression contained the subexpression (a = '1') and the subexpression (a = '0'), coverage required that both subexpressions be true at once. Now we descend into these relational subexpressions and discern that both use the variable "a". Also, subexpressions like (a and b) = '1' were not handled correctly. A side effect of this fix is that spurious warnings are generated when you have a relational expression involving an integer variable. That case will not be handled until the next release.

    New Features Added to 6.2c

    • The vsim shell command has added support for standard X11 application switches: 
        -colormap new
  -display display spec
  -geometry geometry spec
  -name name
  -sync
  -visual visual
  --
      Where:
      1. -colormap new specifies that the window should have a new private colormap instead of using the default colormap for the screen
      2. display spec specifies the name of the display to use (e.g. :0)
      3. geometry spec specifies the size and location of the main window in the form WxH+X+Y
      4. name specifies the title to be displayed in the window, and as the name of the interpreter for send commands.
      5. visual specifies the visual to use for the window. Visual may have any of the forms: class depth, default, number, or best ?depth?.
        1. class depth is one of: directcolor, grayscale, greyscale, pseudocolor, staticcolor, staticgray, staticgrey, or truecolor, or a unique abbreviation. depth specifies how many bits per pixel are needed for the visual
        2. default the default visual for the screen
        3. number specifies a visual X identifier
        4. best ?depth? choose the ``best possible'' visual
      Notes:
      1. Support of the -colormap and -visual options are dependent on what colormap and visuals options are supported by the X server.
      2. The -name option will only effect the application name used in the send command. The title is not effected.
    • SDF annotation support of setuphold and recrem timing checks using a combination of explicit condition arguments and implicit conditions on events were added.
    • When vlog is invoked with the -lint option, warnings will be displayed for enum type mismatches between the operands of binary comparison operators.
    • The project compileall command has been enhanced to print the set of compile commands without executing the compilations. This feature is enabled by using the -n option. For example:
      project compileall -n
    • The -recursive switch is added with vcover merge -du utility, to merge hierarchies below the given design unit. This -recursive switch is valid only when the -du switch is specified.
    • SystemC variables of user-defined types, user-defined channels and ports can now be debuggable using the custom debug interface. Use the custom debug registration macros SC_MTI_REGISTER_CUSTOM_DEBUG and SC_MTI_REGISTER_NAMED_CUSTOM_DEBUG for registering SystemC objects for custom debug. Refer to the section "Custom Debug Interface for Channels and Variables" in the User's Manual for more information.
